The way we work has changed

You now reach for an agent to analyze data, write transformations, and generate reports. But Excel — the tool that holds most of the world's business logic — was never designed for this.

An agent thrives on explicit structure. Excel thrives on flexible ambiguity. That gap isn't a minor annoyance — it's a fundamental impedance mismatch that turns your spreadsheet from an asset into a bottleneck.

Let's walk through exactly why, with code you can run right here in the browser.

1. No structural contract

Binary formats aren't the enemy — Parquet, Arrow, and SQLite are binary, and agents work with them effortlessly. The problem is that Excel has no schema, no contract, no guarantees about what's inside.

Is row 3 a header? Are columns B through D merged? Is that number a raw value or the result of a formula that references a hidden sheet? The agent has to discover the structure through trial and error — every single time.

A Parquet file declares its columns, types, and statistics in a footer. A CSV file is plain text with a simple convention. An Excel file is a ZIP of XML where data, formatting, formulas, and layout are interleaved with no separation of concerns.

2. Schema by accident

Excel infers types from values. A column is "mostly numbers except row 47 has N/A." The agent has to discover this through trial and error rather than being told: price: number | null.

In a database or a typed DataFrame, the schema is explicit. The agent knows exactly what it's working with before it runs a single operation. Zero guesswork.

3. GUI-first, data-second

Merged cells. Conditional formatting. Freeze panes. Pivot tables. Named ranges that may or may not exist. All of these encode presentation logic — visual layout for human eyes — interleaved with the actual data.

An agent has to untangle what's data from what's decoration. In code, data and presentation are separate layers by design. The agent only touches the data layer.

4. Positional fragility

=VLOOKUP(A2, Sheet2!B:D, 3, FALSE) breaks the moment someone inserts a column before D. The logic is coupled to coordinates, not names.

This is the same reason we moved from numbered registers to named variables in programming decades ago. An agent writing JOIN ON product_id doesn't care what column number anything is — the name is the contract.

5. Stateful, hidden computation

Excel's reactive model means a formula's dependencies are implicit. Circular references, iterative calc, volatile functions like RAND() — the agent can't statically analyze a workbook. It has to run it.

A Python script or SQL query makes data flow explicit. Input → transform → output. Every step is visible, named, and composable. An agent can reason about the pipeline without executing it.

The better way

The pattern is straightforward:

  • Separate data from logic — CSV, JSON, or Parquet files hold values only. No formulas, no formatting, no merged cells.
  • Transform with code — SQL queries, Python/pandas, or your language of choice. Explicit, testable, composable pipelines.
  • Visualize as a rendering layer — notebooks, dashboards, charts. Swap the viz without touching the data or the logic.

Each layer is independently readable, testable, and composable. An agent can work at any layer without untangling the others. Excel smears all three into one file — that's the bottleneck.

The ecosystem that replaced Excel

In 2007, Wes McKinney was a quant at AQR Capital Management. Complex models took 30 minutes to discuss — but months to implement in Excel and SQL. Frustrated, he built a tool to make data analysis fast and composable. That tool became pandas.

pandas didn't just make Python viable for finance — it sparked an ecosystem of composable tools, each doing one thing well:

  • NumPy — fast array math
  • pandas — labeled DataFrames, joins, groupby
  • Apache Arrow — zero-copy in-memory data (also McKinney)
  • Polars — GPU-accelerated, lazy evaluation
  • DuckDB — embedded SQL analytics engine

The key insight: these tools are composable. CSV → pandas → Arrow → DuckDB. Each step is a plain-text artifact or an in-memory table. An agent can inspect, modify, or replace any link in the chain. The opposite of a monolithic .xlsx.

Excel isn't the enemy

Excel is a phenomenal tool for humans exploring data interactively. The problem arises when it becomes the persistence layer for business logic in an agent-driven workflow.

The fix isn't to stop using Excel entirely. It's to use it as an output format — generate the .xlsx at the end for stakeholders who need it. Keep your source of truth in plain text, your transformations in code, and your visualizations as a separate concern.

Your agents will thank you.
