What if I don't like Python?

The pattern isn't about Python — it's about separating data from logic. That works in any language. Here's what pd.read_csv() + groupby() looks like everywhere else.

This presentation uses Python and DuckDB because they're the most common tools for data work. But the core insight — keep data in files, logic in code, and use version control — works in every language. If you're already comfortable with something else, start there. The agent will meet you where you are.

Read a CSV, group by a column, sum a value

Same operation, eight languages. The pattern is identical: load data → transform → output. No VLOOKUPs, no merged cells, no hidden formulas.

🐍 Python

The data science default. pandas + DuckDB + Polars + Arrow. Largest ecosystem.

import pandas as pd
df = pd.read_csv("sales.csv")
df.groupby("region").revenue.sum()

🟨 JavaScript / TypeScript

Full-stack data. DuckDB-WASM runs in the browser. Observable, Arquero, Danfo.js.

import * as duckdb from '@duckdb/duckdb-wasm';
const db = await duckdb.create();
await db.registerFileText("sales.csv", text);
db.query("SELECT region, SUM(revenue)
          FROM sales GROUP BY region");

🔷 C# / .NET

LINQ shines here. DataFrame via Microsoft.Data.Analysis. Polars has .NET bindings.

using Microsoft.Data.Analysis;
var df = DataFrame.LoadCsv("sales.csv");
var result = df.GroupBy("region")
    .Sum("revenue");

🦀 Rust

Blazing fast. Polars was built in Rust. DataFusion, Apache Arrow native bindings.

use polars::prelude::*;
let df = CsvReader::from_path(
    "sales.csv")?.finish()?;
let result = df.groupby(["region"])?
    .agg()?;

🔵 Go

Simple, fast, single binary. Gota for DataFrames. DuckDB has Go bindings.

import "github.com/marcboeker/go-duckdb"
db, _ := sql.Open("duckdb", "")
db.Exec("CREATE TABLE sales AS
  SELECT * FROM 'sales.csv'")
db.Query("SELECT region,
  SUM(revenue) FROM sales
  GROUP BY region")

🟪 PowerShell

Built into Windows. Great for sysadmins who already live in the terminal.

$data = Import-Csv sales.csv
$data | Group-Object region |
  ForEach-Object {
    [PSCustomObject]@{
      Region = $_.Name
      Total = ($_.Group |
        Measure-Object revenue
        -Sum).Sum
    }
  }

📊 R

The statistics gold standard. Tidyverse makes data pipelines read like sentences.

library(dplyr)
df <- read_csv("sales.csv")
df |>
  group_by(region) |>
  summarise(
    total = sum(revenue))

🗄️ SQL (direct)

The universal data language. DuckDB CLI reads CSVs directly — no import step.

-- duckdb CLI, one command:
SELECT region,
       SUM(revenue) AS total
FROM 'sales.csv'
GROUP BY region
ORDER BY total DESC;

The real point

Every example above does the same thing. The language doesn't matter. What matters is the pattern:

📄

Data stays in files — CSV, Parquet, JSON, whatever.

⚙️

Logic lives in code — pick your language, pick your library.

🔄

Version control tracks everything. No more Final_v3_FINAL.xlsx.

🤖

Agents work with all of them. The separation is what makes agents effective.

An agent doesn't care if you write Python or Go or Rust. It cares that your data isn't interleaved with formatting, that your logic is explicit, and that every change is tracked. Start with the language you already know.
